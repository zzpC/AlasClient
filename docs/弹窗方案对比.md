# 弹窗显示逻辑方案对比

为了实现“首次启动弹窗，后续不重复弹窗”的需求，我们探讨了两种核心方案。以下是它们的详细对比。

## 方案一：基于时间戳的“活跃会话” (Time-based Session)
使用 `SharedPreferences` 记录最后一次操作的时间戳，通过判断当前时间与该时间戳的差值来决定是否弹窗。

### 核心原理
- **点击启动时**：在本地存储记录 `last_launch_time = now`。
- **再次进入时**：计算 `now - last_launch_time`。如果小于设定阈值（如 10 分钟），则直接跳转。

### 优缺点
| 维度 | 特点 |
| :--- | :--- |
| **优点** | 状态持久化，即使应用进程被杀掉，只要在时间范围内，依然不弹窗。 |
| **缺点** | 逻辑相对复杂（需处理时间差）；阈值难以界定（设短了可能频繁弹窗，设长了可能导致用户想重新设置时进不去）。 |
| **适用场景** | 希望跨进程生命周期保持“免打扰”状态。 |

---

## 方案二：基于进程存活的“静态变量” (Process-based Variable)
利用 Android 进程内存特性，使用类静态变量（Static Member）记录是否已显示过。

### 核心原理
- **类定义中**：声明 `private static var hasShownInThisSession = false`。
- **首次进入**：变量为 `false`，显示弹窗，操作后设为 `true`。
- **再次进入**：只要进程（App）没被系统或用户彻底关闭，变量在内存中保持 `true`，直接跳转。
- **进程结束**：用户强杀应用或系统回收，内存清空。下次启动变量回到 `false`。

### 优缺点
| 维度 | 特点 |
| :--- | :--- |
| **优点** | **完全符合“App还活着就不弹窗”的直觉**；实现极简，性能最高；无需清理磁盘数据。 |
| **缺点** | 无法跨进程生命周期。如果用户手动杀掉 App 后立即重开，会再次看到弹窗（这通常也是用户预期的行为）。 |
| **适用场景** | **推荐方案**。最符合调试习惯：只要没关浏览器（或者没强杀 App），就不被打扰。 |

---

## 总结与建议

| 特性 | 方案一 (时间戳) | 方案二 (静态变量) |
| :--- | :--- | :--- |
| **存储位置** | 磁盘 (XML文件) | 内存 (RAM) |
| **存活周期** | 固定时长 (如 10 分钟) | 进程生命周期 |
| **强杀App后行为** | 在时间内仍不弹窗 | **立即恢复弹窗** |
| **代码复杂度** | 中 | **低** |

**最终选择：方案二 (静态变量)**
因为它能精准捕捉“App 还没被杀掉”的状态。只要您的 WebView (浏览器) 还在后台运行，或者您只是暂时切出应用，静态变量就会帮您挡住重复的弹窗；而当您通过多任务界面划掉应用（强杀）时，它又能灵敏地感知到“新一轮调试开始”，从而重新弹出设置界面。
